# Claude Code notes

This is a forge-generated project. For orientation and command reference, read
[AGENTS.md](AGENTS.md) first — it captures the generator-level structure and
conventions that apply regardless of the language(s) you're editing.

## Quick orientation

- Run tests per-service: `cd services/<name>` then the language-native test
  command (`uv run pytest`, `npm test`, `cargo test`).
- Start the stack: `docker compose up --build` from the project root.
- See what forge turned on: `cat forge.toml` — `[forge.features].enabled` lists
  every opt-in feature fragment applied to this project.

## Editing tips

- **Prefer editing source, not generated docs**: files in `services/*/README.md`
  and per-feature docs under `docs/` are safe to edit; anything with a
  `Generated by forge — do not edit by hand` header is metadata.
- **Don't reorder middleware** in `services/<python>/src/app/main.py` without
  checking `forge.toml [forge.features]`. Feature fragments register themselves
  via `# FORGE:MIDDLEWARE_REGISTRATION` markers and their order reflects
  Starlette's stack semantics (later-added = outer).
- **New CRUD entity**: add to `backends.<name>.features` in your forge config
  and re-run forge, then verify the migration under `services/<name>/alembic/`
  (Python) or `services/<name>/migrations/` (Rust).

## When something is unexpectedly missing

If you expected a middleware, route, or tool and don't see it, check:

1. `forge.toml` — is the feature listed under `[forge.features].enabled`?
2. `forge/templates/_fragments/<feature>/<language>/` in the generator repo —
   does an implementation exist for this backend?
3. `--list-features` — did the feature get deleted or renamed between forge
   versions?

The answer is usually (1): the feature wasn't on when this project was first
generated. Re-run forge with `features: { <key>: { enabled: true } }` to add
it.
