# Agents playbook

This project was scaffolded by [forge](https://github.com/cchifor/forge). The layout
follows forge conventions; treat `forge.toml` as ground truth for what was
generated and which features are active.

## Layout

- `services/` — one directory per backend service (`services/<name>/`).
  Each is an independent app with its own `pyproject.toml` / `package.json` /
  `Cargo.toml` and runs in its own container.
- `apps/frontend/` — the SPA (Vue / Svelte) or native app (Flutter) when a
  frontend was requested.
- `infra/` — auth infrastructure (Keycloak realm, Gatekeeper) when auth was
  requested.
- `tests/e2e/` — Playwright end-to-end tests (when a frontend is present).
- `docker-compose.yml` — the full stack; `init-db.sh` provisions per-service
  Postgres databases.
- `forge.toml` — generator metadata: forge version, per-template paths, and
  the set of enabled features.

## Common commands

Per-backend (run inside each service directory):

- Python: `uv sync`, `uv run pytest --no-cov -q`, `uv run ruff check src/`.
- Node: `npm install`, `npm test`, `npx biome check src/`.
- Rust: `cargo build`, `cargo test`, `cargo clippy --all-targets -- -D warnings`.

Full stack: `docker compose up --build` from the project root.

## Making changes

- **Business logic**: edit the relevant service under `services/<name>/`.
  Keep routes thin; push logic into `services/` / `domain/` layers.
- **Cross-service shared code**: forge does not generate a shared monorepo
  package. Duplicate small utilities or factor them into a separate library.
- **Docker / infra / ports**: settings are rendered from the original forge
  config. Prefer editing and re-running forge where possible; if you must
  edit by hand, record the divergence at the top of `forge.toml`.
- **Adding a feature after-the-fact**: a future `forge update` command will
  apply a new feature to an existing project; for now, re-run forge with the
  updated `features:` YAML and diff the result against the tracked project.

## What forge maintains

The generator owns:

- The *shape* of each service (directory layout, framework choice, auth
  integration).
- Feature fragments listed under `[forge.features]` in `forge.toml`.

The generator does **not** own:

- Your actual business logic, database schema, or UI.
- Any file you edit that's marked with `# Generated by forge — do not edit`
  at the top, which is tracking metadata rather than behavior.

When regenerating, forge preserves files it never emitted (your own code) and
re-applies feature fragments idempotently.
